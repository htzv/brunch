# 7. Open PRs

One PR per repo. We use `gh` from inside each worktree via `foreach`:

```bash
brunch foreach -- gh pr create --fill --base main
```

`--fill` uses the most recent commit's subject and body for the PR
title/description; `--base main` is explicit (helpful if your repos
have non-default base branches).

Sample output:

```text
==> backend
Creating pull request for task-1234-add-export into main in acme/backend
https://github.com/acme/backend/pull/142

==> frontend
Creating pull request for task-1234-add-export into main in acme/frontend
https://github.com/acme/frontend/pull/87

foreach  task-1234-add-export
  OK     backend   (acme/backend) (exit 0)
  OK     frontend  (acme/frontend) (exit 0)
```

You now have:

- `acme/backend#142` — *feat: add /api/v1/users/export CSV endpoint*
- `acme/frontend#87` — *feat(users): add 'Download CSV' button*

…each reviewable independently, with the same branch name so they're
trivially cross-linkable in PR descriptions or Linear comments.

## Linking the PRs back to the task

This part is convention, not brunch. The simplest pattern: put both PR
URLs in the Linear ticket, and in each PR's description, link the
other PR with a brief one-liner ("Frontend half: acme/frontend#87").
That gives reviewers the cross-repo context without any tooling.

A more ambitious pattern is to use `gh pr edit --body-file` with a
generated body that lists every sibling PR. The brunch design doc
parks "auto-linked PR descriptions" as a possible iteration-3 polish;
for now, a human note works fine.
